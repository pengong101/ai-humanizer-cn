# 智能体架构与工作流 v3.0

**版本：** v3.0  
**确立：** 2026-03-28（v1.0）→ 2026-04-03（v3.0）  
**原则：** Research ↔ 执行 Agent 直接协作，main 只布置任务和汇报  

---

## 一、核心改进（v2.0 → v3.0）

### v2.0 星型结构的问题

```
main ──→ Research ──→ main ──→ Content ──→ main ──→ Review
              ↑                                    ↑
              └──────────────────────────────────────┘
                        （验证阶段才连接，缺乏迭代）
```

**问题：**
1. main 成为瓶颈（每次都要经过）
2. Agent 间缺乏直接协作
3. 迭代效率低
4. 交接材料不明确

### v3.0 协作迭代结构

```
用户需求 → main（布置任务）
              │
              ├─────────────────────────────────────┐
              │                                     │
              ▼                                     ▼
        ResearchAgent ◄──────────► ContentAgent
              │                        │
              │ 提供素材 ─────────────→ 申请素材
              │ 反馈建议 ◄──────────── 接受建议
              │                        │
              └────── 迭代直到达标 ─────┘
                          │
                          ▼
                    ReviewAgent
                          │
                   ┌──────┴──────┐
                   ▼             ▼
                通过          不通过（修改，最多3轮）
                   │             │
                   ▼        3轮未过 → main 决策
                发布
```

## 二、核心原则

### 1. main 的明确边界

| 介入 | 不介入 |
|------|--------|
| 布置任务 | 创作过程 |
| 3轮未过的决策 | 素材选择 |
| 用户需求变更 | 修改细节 |
| 最终汇报 | Agent 间协商 |

### 2. Agent 直接协作

- ResearchAgent 和 ContentAgent/CodeAgent/OpsAgent **直接通信**
- 不需要经过 main 中转
- 主动通知机制：完成时主动通知对方

### 3. 明确的交接材料

每个交接必须有**标准化的材料格式**：
- 素材包（Research → Content）
- 技术调研包（Research → Code）
- 作品提交（Content → Review）
- 技能提交（Code → Review）
- 审核反馈（Review → Agent）

### 4. 迭代与升级

- 审核未通过 → 迭代修改（最多3轮）
- 3轮未过 → 升级 main 决策
- 重大故障 → 立即升级

## 三、四大工作流

### 工作流1：内容创作

```
触发：用户需求 或 06:00 cron
    ↓
main 布置任务
    ↓
ResearchAgent → ContentAgent（直接协作）
    │ 提供素材 ←────────────── 申请素材
    │ 反馈建议 ←──────────── 接受建议
    ↓
迭代直到 ContentAgent 自评分 ≥ 7
    ↓
ReviewAgent 审核
    ↓
┌─ 通过 → 发布 → main 汇报用户
└─ 不通过 → 迭代（最多3轮）→ 3轮未过 → main 决策
```

**触发 ReviewAgent 审核的标准：** 自评分 < 7

### 工作流2：技能研发

```
触发：用户需求
    ↓
main 布置任务
    ↓
ResearchAgent → CodeAgent（直接协作）
    │ 调研方案 ←────────────── 申请技术建议
    │ 对比分析 ←──────────── 提供实现思路
    ↓
迭代直到 CodeAgent 确认方案
    ↓
CodeAgent 开发
    ↓
ReviewAgent 审核（创新性≥9，安全≥9）
    ↓
┌─ 通过 → CodeAgent 发布 → main 汇报用户
└─ 不通过 → 迭代（最多3轮）→ 3轮未过 → main 决策
```

### 工作流3：雷达日报（简化流程）

```
触发：每日 09:00 cron
    ↓
main 布置任务（声明格式）
    ↓
ResearchAgent 搜集情报（24小时）
    ↓
ContentAgent 生成日报
    ↓
自评分 ≥ 7？
    ├─ 是 → 直接发布 → main 汇报（如有异常）
    └─ 否 → ReviewAgent 审核
```

**日报不走完整审核流程，因为是标准化重复任务**

### 工作流4：运维故障

```
触发：cron / heartbeat / 用户报告
    ↓
OpsAgent 初步诊断
    ↓
已知问题？
    ├─ 是 → 自动修复 → 记录 → main 汇报
    └─ 否 → ResearchAgent 调研
                ↓
        找到方案？
            ├─ 是 → OpsAgent 修复
            └─ 否 → 升级 main 决策
                        ↓
                用户决策（继续/暂停/回滚）
                        ↓
                OpsAgent 执行
                        ↓
                修复完成 → main 汇报
```

## 五、自适应信任（保持）

| 复杂度 | 自主权 | 介入 |
|--------|--------|------|
| simple | 高信任 | 完成即报告 |
| medium | 中信任 | 关键节点汇报 |
| complex | 低信任 | 每步确认 |

## 六、工作流选择

| 任务类型 | 工作流 | 审核 |
|---------|--------|------|
| 科普文章/内容创作 | 工作流1 | ReviewAgent 审核（自评分<7时） |
| 技能/SKILL研发 | 工作流2 | ReviewAgent 审核（必须） |
| 雷达日报 | 工作流3 | 简化流程，异常才审核 |
| 运维故障 | 工作流4 | 自动处理为主，升级才介入 |
| 调研任务 | 快速版本 | 直接 Research → main 汇报 |
