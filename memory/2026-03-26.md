## 🔴 重要教训：Bind Mount 架构理解

### 核心发现

**我之前的诊断完全错误！** Bind mount 是正常工作的。

**正确理解：**
```
容器内 /root/.openclaw/workspace/  → btrfs bind mount → 真实 NAS 存储 ✅
容器内 /data_s001/...              → overlay filesystem → 不是真实 NAS ❌
```

**错误原因：**
- 从容器内 exec 检查 `/data_s001/` 看到的是 overlay upperdir 的临时视图
- 不是真实宿主机文件系统
- 我误以为文件没有穿透，实际上已经穿透了

**正确检查方法：**
- 通过 `/root/.openclaw/workspace/` 检查（bind mount 路径）
- 而非 `/data_s001/`（overlay 内部路径）

**验证测试：**
- 用户在 NAS 文件管理器直接访问 `/data_s001/.../workspace/`
- 能看到容器内写入的所有文件
- 包括 radar-reports、articles、clawhub-packages

**结论：**
- Bind mount 完全有效，无需同步脚本
- `/root/.openclaw/` 是真实宿主机映射
- `/data_s001/` 在容器内是 overlay 视图，不是真实路径

## SearXNG 修复（19:00）

### 问题现象
- 所有搜索引擎报 403 Forbidden 或 Name or service not known

### 根因
1. **403 根因**：`settings.yml` 中 `formats` 只允许 `html`，不允许 `json`
2. **Name or service not known**：`clash` 主机名在 openclaw-searxng 网络无法解析

### 解决方案
1. 创建命名卷 `searxng-config`
2. 启动临时容器让配置生成
3. 修改 `settings.yml` 添加：
   - `trust_localhost: true`
   - `trusted_networks: ["172.28.0.0/16", "127.0.0.1", "::1"]`
   - `formats: [json, html]`（关键！）
4. 复制 `limiter.toml`
5. 使用命名卷启动正式容器

- Baidu 引擎正常工作 ✅

## SearXNG 代理配置修复（20:00）

### 问题
searxng-auto-proxy 适配器选择最优节点，但没有让 SearXNG 实际使用代理

### 解决方案
1. **添加 outgoing proxies 配置**到 settings.yml:
```yaml
outgoing:
  proxies:
    all://:
      - http://clash:7890
```

2. **网络问题**：SearXNG 和 Clash 不在同一网络
   - SearXNG 在: `openclaw-searxng`
   - Clash 在: `openclaw_openclaw-net` + `bridge`
   - 解决：`docker network connect openclaw_openclaw-net searxng`

### 测试结果
| 引擎 | 结果数 | 状态 |
|------|--------|------|
| Baidu (直连) | 9 | ✅ 可用 |
| Bing (代理) | 10 | ✅ 可用 |
| Google (代理) | 0 | ❌ access denied |
| Brave (代理) | 0 | ❌ timeout |
| DuckDuckGo (代理) | 0 | ❌ timeout |

### 关键教训
- 适配器只做"选择节点"，不自动更新 SearXNG 配置
- 需要手动添加 `outgoing.proxies` 配置
- 网络连通性需要单独检查和修复
