**版本：** v1.0  
**创建时间：** 2026-03-27  
**灵感来源：** 文章写作工作流 v2.0  
**许可证：** MIT

---

## 核心设计原则

1. **无实际代码不创建 Skill** — 避免空壳工程
2. **无真实问题不创作** — 必须解决用户的实际问题
3. **无创新性不发布** — 差异化价值说明
4. **测试即验证** — 每个功能必须有对应测试
5. **发布即固化** — 一次完整流程，一次性提交

## 五阶段工作流

### 阶段一：需求分析 Researcher

**输入：** 用户需求或问题描述  
**输出：** 需求文档

**分析内容：**
- 解决什么问题？用户是谁？
- 现有解决方案有哪些？差距在哪？
- 差异化价值是什么？
- 核心功能清单（不超过 5 个）
- 拒绝清单（明确不做什么）

**输出格式：**
```markdown
# 需求文档 - [Skill名称]

## 问题陈述
[具体问题]

## 用户画像
[谁会用？]

## 差异化价值
[与现有方案的差距]

## 核心功能（≤5个）
1. [功能名]：[描述]
2. ...

## 拒绝清单
- [不会做的功能]
```

**通过标准：** CEO 审批差异化价值 ≥ 7/10

---

### 阶段二：开发 DevAgent

**输入：** 需求文档  
**输出：** 可运行 Skill

**开发原则：**
- 一个 Skill 一个目录
- 所有功能必须有测试覆盖
- 代码即文档（docstring 完整）
- 禁止硬编码凭据

**必须包含的文件：**
```
[skill-name]/
├── SKILL.md              # 技能描述（强制）
├── [功能实现].py           # 主代码（强制）
├── requirements.txt        # 依赖（强制）
├── README.md               # 快速上手（强制）
├── tests/
│   ├── __init__.py
│   └── test_[skill].py   # 测试（强制）
├── CHANGELOG.md           # 更新日志
└── clawhub.json          # 发布元数据
```

**禁止：**
- `tests/` 目录不存在或为空
- 虚假性能指标（95% 准确率等）
- 不存在文件在 SKILL.md 中列出
- 示例代码无法运行

---

### 阶段三：QC 审核员

**模型：** MiniMax-M2.7

**四维审核（每项 0-10，总分 40）：**

| 维度 | 内容 | 通过线 |
|------|------|--------|
| 合规安全 | 无硬编码凭据、权限最小、依赖无已知漏洞 | ≥8 |
| 代码质量 | 可运行、测试通过、无死代码 | ≥8 |
| 文档质量 | SKILL.md 准确、示例可运行 | ≥8 |
| 创新价值 | 真实差异化、有实际用户价值 | ≥8 |

**评分标准：**
- ✅ **通过**：总分 ≥ 35，且每维 ≥ 8
- 🔄 **修改**：总分 28-34 → 返回开发者
- ❌ **拒绝**：总分 < 28 或任一维 < 6 → 重新需求分析

**QC 报告格式：**
```markdown
# QC 报告 - [Skill名称]

## 四维评分

| 维度 | 得分 | 状态 |
|------|------|------|
| 合规安全 | X/10 | ✅/🔄 |
| 代码质量 | X/10 | ✅/🔄 |
| 文档质量 | X/10 | ✅/🔄 |
| 创新价值 | X/10 | ✅/🔄 |
| **总分** | **XX/40** | **✅/🔄/❌** |

## 问题清单
### 🔴 严重（必须修复）
1. [问题] → [修复建议]

### 🟡 一般
1. [问题] → [建议]

## 最终判定
```
✅ 发布
🔄 修改后发布
❌ 重新需求分析
```
```

### 阶段四：整合 Publisher

**输入：** QC 通过的代码 + 文档  
**输出：** GitHub Release + ClawHub 发布

**发布检查清单：**
- [ ] GitHub Release 创建（版本号 vX.Y.Z）
- [ ] tar.gz 发布包生成
- [ ] clawhub-packages/ 目录更新
- [ ] MEMORY.md 更新（记录发布的 Skill）
- [ ] SKILL.md 元数据更新（version, updated 日期）

**发布包命名：**
```
[skill-name]-latest.tar.gz     # 始终指向最新
[skill-name]-vX.Y.Z.tar.gz    # 版本归档
```

---

### 阶段五：固化 Learning

**每次发布后记录：**
- 解决的问题 vs 预期
- 实际用户反馈（如有）
- 下次迭代改进点

## 创作原则（写给所有智能体）

**收到 Skill 创作指令时，先问自己：**
1. 这个问题用户真的遇到了吗？还是在造轮子？
2. 我的实现和现有 Skill 有没有本质差别？
3. 有没有测试验证核心功能？
4. 发布后用户会真的用吗？

**回答全是"是"才继续。**

## 版本规范

- MAJOR.MINOR.PATCH
- MAJOR：不兼容的 API 变更
- MINOR：新增功能（向后兼容）
- PATCH：Bug 修复

## 示例流程

**用户需求：** "我想创建一个定时提醒技能"

```
CEO：审批 → 通过（PASS）
Researcher：需求分析 → 输出需求文档
DevAgent：开发 → 输出 skill 代码
QC：审核 → 35分通过 ✅
Publisher：发布 → GitHub Release
固化 → MEMORY.md 更新
```
